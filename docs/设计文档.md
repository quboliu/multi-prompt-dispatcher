# Multi-Web-LLM Prompt Dispatcher

**一次输入 · 多模型网页端并行交互插件**
**需求设计与技术方案文档**

---

## 1. 项目背景与核心动机

### 1.1 背景问题

在实际使用中，用户往往同时登录多个 Web 端大模型，例如：

- ChatGPT
- Claude
- Gemini
- AI Studio 等

当面对同一个问题时，现有使用方式存在明显低效点：

- 同一条 prompt 需要在多个网页中重复复制 / 粘贴
- 多轮追问时重复劳动更加明显
- 多模型结果难以系统性对比

---

### 1.2 核心设计目标（重新明确）

> **提供一种安全、可控、符合浏览器与平台边界的方式，让用户“一次决定 prompt，多处执行”。**

强调三点：

1. **prompt 的决策永远来自人类**
2. **执行发生在各模型的官方 Web 页面中**
3. **插件不替代模型，只增强操作效率**

---

### 1.3 明确的非目标（设计边界）

本项目**明确不做**：

- API 调用或请求伪造
- cookie / token 管理
- 后端代理或中转
- 自动生成 prompt
- 统一/融合不同模型的上下文语义

---

## 2. 核心使用哲学（非常重要）

### 2.1 正确理解“统一”

> **统一的是“发送动作”，不是“上下文状态”。**

- 你可以：

  - 单独和 Claude 深聊 10 轮
  - 再回到插件
  - 发起一条新的统一 prompt
- 插件不会：

  - 试图让其他模型“理解 Claude 的上下文”
  - 合并或同步对话历史

这是**合理的、真实的、人类的使用方式**。

---

### 2.2 正确理解“追问”

> **统一追问**  **=**  **把同一条新 prompt，并行追加到多个模型“各自当前的对话线程”中。**

- 每个模型：

  - 有自己独立、真实、原生的上下文
- 插件：

  - 不关心上下文内容
  - 只负责把 prompt 放进去并点击发送

---

## 3. 总体技术方案

### 3.1 技术路线选择

**浏览器插件（Extension）+ Content Script + DOM 操作**

这是在以下约束下的最优解：

- 不破坏安全模型
- 不接触认证信息
- 不逆向请求协议
- 可持续维护

---

## 4. Phase 1：最小可用方案（核心价值阶段）

### 4.1 Phase 1 目标

> **解决“同一个 prompt 不想复制 N 遍”的核心痛点。**

---

### 4.2 功能范围

#### 支持功能

- 统一 prompt 输入
- 自动识别当前已打开的模型页面
- 多模型并行发送（含首次与追问）
- 支持“部分模型参与”

#### 明确支持的使用场景

- 首次统一提问
- 多轮统一追问
- 中途单独与某模型聊天后，再统一发送

---

### 4.4 技术实现要点

#### Adapter 抽象

```ts
interface ModelAdapter {
  detect(): boolean
  sendPrompt(prompt: string): Promise<void>
}
```

#### 核心动作（每次发送）

1. 定位输入区域
2. 设置 prompt 内容
3. 触发发送
4. 不读取、不修改上下文

### 4.5 Phase 1 风险与限制

- 依赖 DOM 稳定性
- 无统一结果视图
- 无失败重试机制（可后续补）

## 5. Phase 2：影子镜像指挥中心（Shadow Dashboard）

### 5.1 Phase 2 目标 

> **打造一个高颜值的“数字化指挥部”页面，以 $m \times n$ 网格布局实时呈现在各后台标签页中发生的对话。**

强调两点：
1. **视觉统一**：摆脱各平台杂乱的 UI 干扰，只显示内容。
2. **实时同步**：看到流式输出的“影子”在 Dashboard 中实时跳动。

---

### 5.2 核心展示架构

```
┌──────────────────────────────────────────────────────────┐
│  Universal Command Center (Dashboard Page)               │
│  ┌──────────────────────┐  ┌──────────────────────┐      │
│  │ [✓] Prompt Input Area  │  │ [ Layout: 2 x 2 ]    │      │
│  └──────────────────────┘  └──────────────────────┘      │
│                                                          │
│  ┌──────────────────────┐  ┌──────────────────────┐      │
│  │ Card: ChatGPT        │  │ Card: Claude         │      │
│  │ [Shadow Mirroring...]│  │ [Shadow Mirroring...]│      │
│  └──────────────────────┘  └──────────────────────┘      │
│  ┌──────────────────────┐  ┌──────────────────────┐      │
│  │ Card: Gemini         │  │ Card: (Waiting...)   │      │
│  │ [Shadow Mirroring...]│  │                      │      │
│  └──────────────────────┘  └──────────────────────┘      │
└──────────────────────────────────────────────────────────┘
```

---

### 5.3 影子镜像（Shadow Mirroring）技术细节

#### A. 输出捕获 (Content Script)
- **监听器**：使用 `MutationObserver` 针对各平台的“回答容器”进行监听。
- **解析层**：
  - 实时提取最新一条回答的 HTML/Markdown。
  - 实时计算生成状态（生成中/已停止）。
- **同步**：将内容通过增量方式发送至 Background，再广播至 Dashboard。

#### B. 渲染引擎 (Dashboard Page)
- **Grid 控制器**：允许用户自定义 $m \times n$ (如 $1\times2, 2\times2, 1\times3$) 布局。
- **高颜值渲染**：使用统一的 Markdown 渲染器（如 `markdown-it`），配合精心设计的 CSS 主题（毛玻璃效果、流畅动画）。
- **性能优化**：对非激活网格进行渲染频率限制。

---

### 5.4 关键交互改进

1. **双向锚点**：点击 Dashboard 中的某个卡片，可直接跳转跳转到浏览器中对应的真实 Tab。
2. **局部刷新**：支持针对单个模型重发 Prompt 或触发其原生的“重新生成”按钮。
3. **沉浸模式**：全屏切换，隐藏所有控制项，纯粹展示对比。

---

### 5.5 Phase 2 风险控制（更新）

- **技术风险**：
  - 各大平台的回答容器 CSS 选择器可能随时失效（需设计 robust 的 Adapter）。
  - 流式数据量大时的通信压力。
- **安全风险**：
  - 数据仅在内存中流动，不进行持久化存储。
  - 完全规避了 Iframe 的安全头限制。

### 6.3 上下文分叉提醒（非常重要）

当出现以下情况时提示用户：

> ⚠ Claude conversation has diverged.
> Unified follow-ups may no longer be semantically equivalent.
