# Claude 适配器深度技术分析

## 1. 编辑器内核：ProseMirror 与复杂 DOM
Claude.ai 的核心输入模块基于 **ProseMirror**，这是一种高度结构化的富文本编辑器，其对输入校验的严格程度超过了 ChatGPT。

### 技术栈画像
- **框架**: 功能复杂的 React 应用，具有严格的 Context 隔离。
- **编辑器**: 非传统的 `div` 或 `textarea`，而是动态解析的文档模型树。

## 2. 交互挑战：从字段边界到事件触发

### 字段边界 (Fieldset Boundaries)
Claude 将输入区域封装在 `fieldset` 标签中。
- **分析**: 这是为了利用浏览器的原生表单管理特性。
- **特性**: 里面的按钮可能带有 `aria-disabled` 属性而非原生 `disabled`，这对探测逻辑提出了更高要求。适配器通过对按钮执行 `getBoundingClientRect` 检查其宽度和高度来确认其真实的视觉存在。

### 执行命令与状态同步
- **核心逻辑**: Claude 对 `input` 事件的监听非常“深”。简单的属性赋值无法触发现端状态索引的更新。
- **解决方案**: 适配器强力依赖 **`document.execCommand`**。
  - `execCommand('selectAll')`：确保旧内容被完全覆盖。
  - `execCommand('insertText')`：这是注入文本最安全的方法，因为它会作为一个原生的用户操作进入 ProseMirror 的事务流（Transaction Flow），从而被 React 捕获。

## 3. 发送机制：轮询与 Enter 键序列
Claude 的发送按钮逻辑极其动态：
- **动态渲染**: 在没有内容时，按钮不仅是 Disabled，有时甚至是“视觉不可用”的。
- **延迟响应**: 设置内容后，按钮的启用状态转化可能需要上百毫秒。
- **应对方案**: 
  - **指数轮询**: 5 次尝试，每次间隔 150ms，以捕捉按钮由灰转蓝的瞬间。
  - **键盘事件序列**: 由于现代前端框架可能监听 `keydown`、`keypress` 和 `keyup` 的组合，适配器模拟了这一完整的物理链条，以确保能成功触发 Enter 发送。

## 4. 网络拦截与数据流保护

### 拦截策略
- **API 路径**: 路径包含 `/organizations/` 动态 ID 和 `/append_message`。
- **拦截逻辑**: 针对 Claude 的 SSE，由于存在 `delta` 和 `completion` 两种报文格式，拦截器执行了“双路径解析”。

### 防回滚与数据清洗
- **特征**: Claude 的流式返回有时会出现“抖动”（例如先返回完整的一段，再返回一个较短的 Delta）。
- **逻辑保护**: `network-interceptor.js` 引入了 **“80% 阈值保护”** —— 如果当前处于生成状态且新报文长度突然缩短至 80% 以下，且更新时间间隔小于 2 秒，则将其视为网络乱序并忽略更新。

## 5. DOM 稳定性评估
- **高危点**: `div.ProseMirror` 内部的 `p` 或 `span` 标签层级经常变动。
- **稳定点**: 适配器专注于 `aria-label` 属性。
- **建议**: 后续适配应考虑利用 `data-testid`（如果存在）或通过输入框与发送按钮的**几何相对位置**进行逻辑锚定。
