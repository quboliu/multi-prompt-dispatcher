# DeepSeek 适配器深度技术分析

## 1. 核心架构特征
DeepSeek 使用了一套现代化的 React 体系，其显著特征是高度的模块化和对用户交互事件的深度劫持。

### 技术栈画像
- **框架**: React + Vite/Webpack
- **样式**: 使用 CSS Modules，类名中包含动态 Hash（如 `_button_1a2b3`），这意味着传统的 CSS 类名选择器极不稳定。
- **状态同步**: 采用了 React 的受控组件模式，对 `input` 事件有严格的拦截处理。

## 2. 交互难点：穿透 React 的受控机制

### React Value Tracker
- **分析**: 在 DeepSeek 这种深度的 React 应用中，仅仅修改 `textarea.value` 会导致 React 内部的 `__valueTracker` 失去同步。当用户尝试搜索或发送时，React 会认为值仍是空的。
- **解决方案**: 适配器在 `setPrompt` 中应用了 **React 注入技术**：
  1. 探测元素的 `_valueTracker` 控制器。
  2. 调用原生 Setter。
  3. 通过 `dispatchEvent` 手动派发一系列合成事件，迫使 React 重新读取 DOM 节点的当前状态。
- **优先级策略**: 优先执行 `execCommand('insertText')`，因为它不仅填入内容，还能触发绝大多数 React 版本内置的更新钩子。

## 3. 发送按钮：启发式探测逻辑
DeepSeek 的发送按钮在样式上与“上传”、“模型切换”按钮极其相似。

### 右侧优先探测法 (Rightmost Detection)
由于类名哈希化，适配器无法通过类名区分功能。它采用了**基于几何位置的启发式算法**：
- **逻辑**: 
  1. 获取输入框容器。
  2. 扫描所有可见按钮。
  3. 执行 `getBoundingClientRect()`。
  4. 排除掉宽/高过小或带典型上传图标（回形针等）的按钮。
  5. 选取 **`rect.right` 最大的元素**。这利用了主流 UI 设计中“发送按钮位于输入框右下角”的规律。

## 4. 网络拦截与 API 分析

### 标准化的利弊
- **路径**: `/api/v0/chat/` (兼容 OpenAI 风格)。
- **解析位**: `choices[0].delta.content`。
- **挑战**: DeepSeek 有时会在响应中夹杂复杂的“推理步骤”（Thought Chain）或特定的引用。拦截器需要通过正则过滤掉某些特殊的报文元数据，仅提取呈现给用户的文本。

## 5. 生成状态与页面响应性
- **生成波段波段**: DeepSeek 的生成通常伴随着停止按钮的出现。
- **探测逻辑**: 适配器通过全局检索包含“停止”文本的 ARIA 标签来实现状态反馈。

## 6. 稳定性建议
- **警告**: 不要依赖 CSS 类名，它们在每次版本发布时都会改变。
- **建议**: 应维持目前的“位置感应 + 关键词排除”方案，这是解决 CSS Module 动态化问题的最稳妥策略。
