# Qwen (通义千问) 适配器深度技术分析

## 1. 核心架构特征
通义千问 (Qwen) 作为阿里云的大模型旗舰产品，其前端工程化程度极高，深度集成了 **Ant Design** 组件体系。

### 技术栈画像
- **框架**: React
- **组件库**: 阿里云定制版 Ant Design
- **输入方式**: 标准 `textarea`，但通过高度定制的样式进行视觉美化。

## 2. 交互难点：穿透组件边界

### 受控组件与 React 状态
- **特征**: Qwen 的输入框是典型的受控组件。简单的 DOM 修改不仅无法生效，反而会被 React 下一次渲染时的虚拟 DOM 对比（Diffing）所覆盖。
- **解决方案**: 
  - **原型劫持 (Prototype Hijacking)**: 适配器采用了一项高级技术 —— 临时修改 `HTMLTextAreaElement.prototype`。通过劫持原生的 `value` Setter，我们可以在脚本设置值时直接触发布持该 DOM 属性的 React 观察者。
  - **双重事件分发**: 除了设置值，必须连续分发 `input` 和 `change` 事件。在 Qwen 的场景下，如果不派发 `change`，其“发送”按钮的状态检测逻辑将无法被激活。

## 3. 发送按钮：Omni UI 的独特性
Qwen 使用了一套被称为 "Omni" 的交互体系。

### 按钮识别逻辑
- **特征**: 发送按钮包含复杂的类名，如 `.omni-button-content-btn`。
- **稳定探测**: 适配器不仅匹配类名，还重点检查了按钮的 **`ant-btn-circle`** 属性。这种基于 Ant-Design 原子类名的探测方案比匹配业务类名更稳定。
- **几何容错**: 在 Qwen 的侧边栏模式（Sidebar Mode）下，UI 结构会大幅收缩。适配器为此设计了“就近原则”，即在输入框所在的 `div` 层级内寻找层级最浅的提交按钮。

## 4. 网络拦截与数据格式

### 阿里云网关协议
- **路径**: `/api/v2/chat/`。
- **响应模式**: 虽然也是流式，但 Qwen 经常在一次 HTTP 响应中通过特殊的换行符分发多个 JSON 块。
- **解析策略**: 拦截器针对 Qwen 采用了“块数据清理（Block Cleaning）”技术，能够识别并跳过多出的元数据字符，确保 `parseResponse` 拿到的是纯净的 JSON。

## 5. 稳定性评估
- **优势**: 由于使用了成熟的 Ant Design 体系，其 DOM 层级非常有规律。
- **风险**: 阿里云经常在“实验版”和“稳定版”之间切换入口（如 `tongyi.aliyun.com` 和 `chat.qwen.ai`）。
- **策略**: 适配器在 `detect()` 中实现了全域名覆盖，并使用同一套选择器逻辑，目前表现稳健。
