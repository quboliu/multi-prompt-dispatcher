# Doubao (豆包) 适配器深度技术分析

## 1. 核心架构特征
豆包 (Doubao) 是字节跳动的核心 AI 产品，其前端深度应用了字节自研的 **Semi Design** 组件库，代表了国内大模型 UI 的顶级工程水平。

### 技术栈画像
- **框架**: React
- **组件库**: Semi Design (字节跳动自研)
- **输入方式**: 高度封装的 `textarea`。

## 2. 交互难点：Semi Design 的封装层

### 穿透组件库事件
- **现象**: Semi Design 的 `Input` 组件在内部对 DOM 原子事件进行了大量过滤和重新封装。直接向 `textarea` 填入 `value` 会导致 UI 上的蓝色聚焦框存在，但模型层检测不到数据。
- **解决方案**: 
  - **`execCommand` 优先法**: 适配器在 `setPrompt` 中优先通过 `document.execCommand('insertText')`。
  - **原理**: 这种方法绕过了属性拦截，直接向浏览器的编辑缓冲区注值，这会强制 Semi Design 的内部监听器感知到变化并触发 React 状态更新。

## 3. 几何传感探测：简洁 UI 下的按钮捕捉
豆包的 UI 风格极度简约（Minimalist），其发送按钮往往是一个没有任何文字信息的圆形图标。

### 按钮“指纹”探测
- **分析**: 传统的文本匹配 (`text="发送"`) 对豆包无效。
- **策略**: 
  1. **层级相关性**: 适配器从输入框元素开始。
  2. **容器溯源**: 向父级追溯 4 层，寻找具有 `aria-label` 或 `role="button"` 的元素。
  3. **位置感应**: 豆包的 UI 设计保证了发送按钮必然位于输入区域的右下角坐标系内。适配器通过对 `boundingClientRect` 的精准偏移计算，在即使类名完全变动的情况下也能锁定按钮。

## 4. 网络拦截与 API 体系

### Bytedance 后端协议
- **路径**: `/api/chat/` 或特定后缀（如 `samantha`）。
- **拦截逻辑**: 豆包的响应报文对 SSE 的遵循非常标准，但在某些实验性对话中会启用一种特殊的压缩格式。拦截器通过监测 `data: ` 包的 JSON 结构，动态判断提取路径（如 `data.text`）。

## 5. 状态机：思考与生成
- **特征**: 豆包在思考时，发送按钮会消失，取而代之的是一个位于底部的“停止”按钮。
- **探测逻辑**: 适配器通过检测全局特殊的类名 `.stop-button` 来同步感知模型的工作状态，从而控制 Dashboard 端的发送频率。

## 6. 稳定性评估
- **优势**: Semi Design 组件库的结构非常稳固。
- **挑战**: 字节经常进行 A/B 测试，不同用户看到的 DOM 结构可能存在细微差异。
- **方案**: 适配器内置了多套选择器数组，通过循环探测实现了对不同实验版本的兼容。
