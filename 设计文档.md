# Multi-Web-LLM Prompt Dispatcher

**一次输入 · 多模型网页端并行交互插件**
**需求设计与技术方案文档**

---

## 1. 项目背景与核心动机

### 1.1 背景问题

在实际使用中，用户往往同时登录多个 Web 端大模型，例如：

- ChatGPT
- Claude
- Gemini
- AI Studio 等

当面对同一个问题时，现有使用方式存在明显低效点：

- 同一条 prompt 需要在多个网页中重复复制 / 粘贴
- 多轮追问时重复劳动更加明显
- 多模型结果难以系统性对比

---

### 1.2 核心设计目标（重新明确）

> **提供一种安全、可控、符合浏览器与平台边界的方式，让用户“一次决定 prompt，多处执行”。**

强调三点：

1. **prompt 的决策永远来自人类**
2. **执行发生在各模型的官方 Web 页面中**
3. **插件不替代模型，只增强操作效率**

---

### 1.3 明确的非目标（设计边界）

本项目**明确不做**：

- API 调用或请求伪造
- cookie / token 管理
- 后端代理或中转
- 自动生成 prompt
- 统一/融合不同模型的上下文语义

---

## 2. 核心使用哲学（非常重要）

### 2.1 正确理解“统一”

> **统一的是“发送动作”，不是“上下文状态”。**

- 你可以：

  - 单独和 Claude 深聊 10 轮
  - 再回到插件
  - 发起一条新的统一 prompt
- 插件不会：

  - 试图让其他模型“理解 Claude 的上下文”
  - 合并或同步对话历史

这是**合理的、真实的、人类的使用方式**。

---

### 2.2 正确理解“追问”

> **统一追问**  **=**  **把同一条新 prompt，并行追加到多个模型“各自当前的对话线程”中。**

- 每个模型：

  - 有自己独立、真实、原生的上下文
- 插件：

  - 不关心上下文内容
  - 只负责把 prompt 放进去并点击发送

---

## 3. 总体技术方案

### 3.1 技术路线选择

**浏览器插件（Extension）+ Content Script + DOM 操作**

这是在以下约束下的最优解：

- 不破坏安全模型
- 不接触认证信息
- 不逆向请求协议
- 可持续维护

---

### 3.2 高层架构

```
┌──────────────────────────────┐
│ Browser Extension UI         │
│ (Popup / Side Panel)         │
│ - Prompt Input               │
│ - Target Selection           │
│ - Result View (Phase 2+)     │
└──────────────┬───────────────┘
               │ message passing
┌──────────────▼───────────────┐
│ Content Scripts (per tab)    │
│ - Adapter detection          │
│ - DOM interaction            │
│ - Output observation         │
└──────────────┬───────────────┘
               │
┌──────────────▼───────────────┐
│ Official LLM Web Pages       │
│ (Unmodified, Logged-in)     │
└──────────────────────────────┘
```

## 4. Phase 1：最小可用方案（核心价值阶段）

### 4.1 Phase 1 目标

> **解决“同一个 prompt 不想复制 N 遍”的核心痛点。**

---

### 4.2 功能范围

#### 支持功能

- 统一 prompt 输入
- 自动识别当前已打开的模型页面
- 多模型并行发送（含首次与追问）
- 支持“部分模型参与”

#### 明确支持的使用场景

- 首次统一提问
- 多轮统一追问
- 中途单独与某模型聊天后，再统一发送

---

### 4.3 用户交互（Popup 示例）

```
┌──────────────────────────────┐
│ Multi-LLM Dispatcher         │
│──────────────────────────────│
│ Prompt:                      │
│ ┌──────────────────────────┐ │
│ │                           │ │
│ └──────────────────────────┘ │
│                              │
│ Targets:                     │
│ [✓] ChatGPT (tab #2)         │
│ [✓] Claude  (tab #5)         │
│ [ ] Gemini                   │
│                              │
│        [ Send to All ]       │
│                              │
│ ⚠ Each model uses its own    │
│   current conversation       │
└──────────────────────────────┘
```

---

### 4.4 技术实现要点

#### Adapter 抽象

```ts
interface ModelAdapter {
  detect(): boolean
  sendPrompt(prompt: string): Promise<void>
}
```

#### 核心动作（每次发送）

1. 定位输入区域
2. 设置 prompt 内容
3. 触发发送
4. 不读取、不修改上下文

---

### 4.5 Phase 1 风险与限制

- 依赖 DOM 稳定性
- 无统一结果视图
- 无失败重试机制（可后续补）

## 5. Phase 2：统一只读结果视图（实用阶段）

### 5.1 Phase 2 目标 

> **让用户在一个视图中，对比同一轮 prompt 的多模型回答。**

---

### 5.2 新增能力

- 捕获模型最终输出（只读）
- Side Panel 展示
- 按轮次组织
- 支持折叠、复制

---

### 5.3 结果视图示意

```
Round 4
────────────────────────
Prompt:
"Can you give a concrete example?"

ChatGPT:
[回答内容...]

Claude:
[回答内容...]

Gemini:
[回答内容...]
```

---

### 5.4 关键设计原则

- **只读**
- 不允许在插件内继续对话
- 深聊必须回到原生页面

---

## 6. Phase 3：对比与认知辅助（高级阶段）

### 6.1 Phase 3 目标

> **从“展示结果”升级为“帮助理解差异”。**

---

### 6.2 可选增强功能

- 关键点抽取
- 回答结构对齐
- 差异提示
- 上下文分叉提示

---

### 6.3 上下文分叉提醒（非常重要）

当出现以下情况时提示用户：

- 某模型被单独追问
- 某模型切换了会话
- 某模型生成失败

提示示例：

> ⚠ Claude conversation has diverged.
> Unified follow-ups may no longer be semantically equivalent.
